
**Penjelasan Komponen Kunci:**

*   **`app/Models`**: Berisi definisi objek data (Model) yang berinteraksi dengan database melalui Eloquent ORM.
*   **`app/Contracts/Repositories`**: Direktori ini berisi antarmuka (interfaces) untuk repository. Ini adalah inti dari Repository Pattern, yang mengabstraksi lapisan persistensi data dari logika bisnis.
*   **`app/Repositories` (diasumsikan ada)**: Implementasi dari antarmuka repository yang akan berinteraksi langsung dengan model Eloquent atau sumber data lainnya. (Tidak ada file .php yang disediakan dalam context, tetapi diasumsikan ada implementasinya).
*   **`app/Services`**: Berisi lapisan layanan (Service Layer) yang mengimplementasikan logika bisnis utama aplikasi. Layanan ini menggunakan repository untuk mengakses dan memanipulasi data.
*   **`app/Policies`**: Mendefinisikan aturan otorisasi untuk model tertentu, memungkinkan kontrol akses yang granular.
*   **`routes/api.php`**: Akan berisi definisi semua endpoint API yang diekspos oleh backend.

## 4. Database yang Digunakan

Proyek ini menggunakan **Eloquent ORM** (Object-Relational Mapper) bawaan Laravel. Eloquent mendukung berbagai sistem manajemen database relasional (RDBMS) seperti:
*   **MySQL** (paling umum)
*   PostgreSQL
*   SQLite
*   SQL Server

Konfigurasi database spesifik akan ditentukan dalam file `.env` dan `config/database.php`. Tanpa file tersebut, diasumsikan dapat menggunakan salah satu dari yang disebutkan di atas, dengan MySQL sebagai pilihan default yang paling sering digunakan dalam proyek Laravel.

## 5. Sistem Otentikasi dan Otorisasi

### Otentikasi
Proyek ini menggunakan **Laravel Sanctum** untuk otentikasi API. Ini terlihat dari penggunaan trait `HasApiTokens` pada model `App\Models\User`. Sanctum memungkinkan otentikasi berbasis token API ringan untuk SPA (Single Page Applications) dan aplikasi mobile.

**Alur Otentikasi Sanctum:**
1.  Pengguna mengirimkan kredensial (misalnya email/username dan password) ke endpoint login API.
2.  Backend memvalidasi kredensial.
3.  Jika valid, backend membuat token API baru untuk pengguna tersebut dan mengembalikannya ke frontend.
4.  Frontend menyimpan token ini (misalnya di `localStorage` atau `sessionStorage`).
5.  Untuk setiap permintaan API berikutnya yang memerlukan otentikasi, frontend menyertakan token ini dalam header `Authorization` (misalnya `Bearer <token>`).
6.  Backend memvalidasi token untuk setiap permintaan untuk memastikan pengguna terotentikasi.

### Otorisasi
Otorisasi dikelola menggunakan **Laravel Policies**. Contohnya adalah `App\Policies\UserPolicy.php` yang mendefinisikan siapa yang berhak `view`, `create`, `update`, `delete` `User` model, berdasarkan peran (`admin`) atau kepemilikan (`user->id === model->id`).

## 6. Potensi Integrasi Firebase di Frontend

**Ya, proyek ini dapat diintegrasikan dengan Firebase Authentication di sisi frontend.**

**Alur Integrasi yang Disarankan:**

1.  **Frontend (Next.js)** menggunakan **Firebase SDK** untuk otentikasi:
    *   **Login Email/Password:** Pengguna login menggunakan Firebase `signInWithEmailAndPassword()`.
    *   **Login Google:** Pengguna login menggunakan Firebase `signInWithPopup(GoogleAuthProvider)` atau `signInWithRedirect()`.
2.  Setelah pengguna berhasil login di Firebase, Firebase akan mengembalikan **Firebase ID Token**.
3.  **Frontend** akan mengambil Firebase ID Token ini.
4.  **Frontend** kemudian mengirimkan Firebase ID Token ini ke backend Laravel melalui endpoint khusus (misalnya `/api/firebase-auth`).
5.  **Backend (Laravel)** akan:
    *   Menerima Firebase ID Token.
    *   Menggunakan **Firebase Admin SDK** (melalui Composer package seperti `kreait/laravel-firebase`) untuk memverifikasi validitas Firebase ID Token.
    *   Setelah token diverifikasi dan pengguna Firebase diidentifikasi, backend akan mencari atau membuat pengguna yang sesuai di database lokalnya (model `App\Models\User`).
    *   Backend kemudian membuat **token Laravel Sanctum** untuk pengguna lokal tersebut.
    *   Backend mengembalikan token Sanctum ini ke frontend.
6.  **Frontend** kemudian menggunakan token Sanctum ini untuk semua permintaan API berikutnya ke backend Laravel, sama seperti otentikasi Sanctum reguler.

**Keuntungan:**
*   Memanfaatkan keamanan dan kemudahan Firebase Authentication.
*   Backend tetap mengontrol sesi API-nya sendiri dengan Sanctum, memisahkan concern otentikasi pihak ketiga dari otentikasi aplikasi internal.

## 7. Alur API Lengkap

Berikut adalah perkiraan alur API berdasarkan Services dan Policies yang ditemukan:

### A. Otentikasi & Profil Pengguna

1.  **Registrasi Pengguna Baru (Asumsi)**
    *   **Endpoint:** `POST /api/register`
    *   **Deskripsi:** Mendaftarkan pengguna baru dan mungkin membuat entri alumni terkait.
    *   **Input:** `username`, `email`, `password`, dan data alumni lainnya.
    *   **Output:** Token Sanctum, data pengguna.
    *   **Service:** (Asumsi ada)
    *   **Policy:** N/A (biasanya publik)

2.  **Login Pengguna (Asumsi)**
    *   **Endpoint:** `POST /api/login`
    *   **Deskripsi:** Mengotentikasi pengguna dan mengembalikan token API.
    *   **Input:** `email`/`username`, `password`.
    *   **Output:** Token Sanctum, data pengguna.
    *   **Service:** (Asumsi ada)
    *   **Policy:** N/A (publik)

3.  **Dapatkan Profil Alumni Berdasarkan User ID**
    *   **Endpoint:** `GET /api/alumni/profile/{userId}` (atau `/api/user/profile` jika terotentikasi)
    *   **Deskripsi:** Mengambil data profil alumni terkait dengan user ID.
    *   **Input:** `userId` (dari URL atau dari token otentikasi).
    *   **Output:** Objek `Alumni` lengkap.
    *   **Service:** `AlumniService::getProfileByUserId`
    *   **Policy:** `UserPolicy::view` (admin atau pemilik profil)

4.  **Perbarui Profil Alumni**
    *   **Endpoint:** `PUT /api/alumni/profile/{userId}` (atau `/api/user/profile` jika terotentikasi)
    *   **Deskripsi:** Memperbarui data pengguna (`users` table) dan data alumni (`alumnis` table). Dapat juga mengelola upload foto profil.
    *   **Input:** `userId` (dari URL atau dari token otentikasi), `name` (untuk `username`), `email`, `password` (opsional), `img_profile` (file), dan data alumni lainnya.
    *   **Output:** Objek `Alumni` yang telah diperbarui.
    *   **Service:** `AlumniService::updateProfile`
    *   **Policy:** `UserPolicy::update` (admin atau pemilik profil)

5.  **Dapatkan Semua Data Alumni**
    *   **Endpoint:** `GET /api/alumni`
    *   **Deskripsi:** Mengambil semua data alumni.
    *   **Input:** N/A
    *   **Output:** Koleksi objek `Alumni`.
    *   **Service:** `AlumniService::getAllAlumni`
    *   **Policy:** `UserPolicy::viewAny` (hanya admin)

### B. Berita / Informasi

1.  **Dapatkan Berita yang Dipublikasikan**
    *   **Endpoint:** `GET /api/news/published`
    *   **Deskripsi:** Mengambil daftar berita yang berstatus publik.
    *   **Input:** N/A
    *   **Output:** Koleksi objek `News` yang dipublikasikan.
    *   **Service:** `NewsService::getPublishedNews`
    *   **Policy:** N/A (biasanya publik)

2.  **Dapatkan Berita Berdasarkan Slug**
    *   **Endpoint:** `GET /api/news/{slug}`
    *   **Deskripsi:** Mengambil detail berita berdasarkan slug uniknya.
    *   **Input:** `slug` dari URL.
    *   **Output:** Objek `News` tunggal.
    *   **Service:** `NewsService::getNewsBySlug`
    *   **Policy:** N/A (biasanya publik)

3.  **Buat Berita Baru**
    *   **Endpoint:** `POST /api/news`
    *   **Deskripsi:** Membuat entri berita baru.
    *   **Input:** `userId` (dari token otentikasi), `title`, `content`, `thumbnail` (file), `is_published` (boolean), dll.
    *   **Output:** Objek `News` yang baru dibuat.
    *   **Service:** `NewsService::storeNews`
    *   **Policy:** Diperlukan middleware `role:admin`.

### C. Sistem Kuesioner (Tracer Study)

1.  **Dapatkan Daftar Kuesioner Aktif**
    *   **Endpoint:** `GET /api/questionnaires`
    *   **Deskripsi:** Mengambil kuesioner yang tersedia untuk diisi oleh alumni.
    *   **Service:** `QuestionnaireRepositoryInterface::getActiveWithQuestions`
    *   **Otentikasi:** Bearer Token (Sanctum).

2.  **Submit Jawaban Kuesioner**
    *   **Endpoint:** `POST /api/questionnaires/submit`
    *   **Deskripsi:** Menyimpan jawaban alumni terhadap pertanyaan kuesioner.
    *   **Input:** `question_id`, `answer_value`, `alumni_id`.
    *   **Service:** (Logika di `QuestionnaireController` atau service terkait).

3.  **Lihat Jawaban Saya**
    *   **Endpoint:** `GET /api/questionnaires/{id}/my-answers`
    *   **Deskripsi:** Melihat riwayat jawaban yang telah dikirimkan oleh user yang sedang login.

### D. Lowongan Pekerjaan (Job Vacancy)

1.  **List Lowongan (Publik)**
    *   **Endpoint:** `GET /api/jobs`
    *   **Deskripsi:** Menampilkan semua lowongan pekerjaan yang tersedia.

2.  **Kelola Lowongan (Admin Only)**
    *   **Endpoint:** `POST /api/jobs`, `POST /api/jobs/{id}`, `DELETE /api/jobs/{id}`
    *   **Deskripsi:** Operasi CRUD untuk admin mengelola informasi lowongan.

### C. Sinkronisasi Firebase & Laravel Sanctum

## 10. Keamanan & Validasi

1.  **Otorisasi Berbasis Policy:** Setiap akses ke resource sensitif dicek melalui `App\Policies`. Misalnya, pengguna tidak bisa mengubah profil alumni orang lain.
2.  **Validasi Request:** Menggunakan `FormRequest` (seperti `LoginRequest`) untuk memastikan data yang masuk ke controller sudah bersih dan sesuai format.
3.  **Sanctum Token Hashing:** Token yang disimpan di database di-hash, sehingga aman jika database terekspos.
4.  **CORS:** Pastikan `config/cors.php` diatur untuk mengizinkan domain Next.js Anda.
